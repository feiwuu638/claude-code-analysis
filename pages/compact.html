记忆与上下文管理

Claude Code 通过三个互补的机制解决 AI 的"失忆"问题：持久化 Memory 文件、会话内 Session Memory、以及上下文超限时的自动压缩（Compact）。

三层记忆体系

层次机制生命周期存储位置
长期记忆Memory 文件（MEMORY.md + 各 .md 文件）跨会话持久项目目录 / ~/.claude/
会话记忆Session Memory（结构化笔记）单次会话内~/.claude/projects/
上下文压缩Compact（自动摘要）上下文超限时触发内存（替换消息历史）

Memory 文件系统

Memory 文件是 Claude Code 的长期记忆载体，在每次会话启动时通过 loadMemoryPrompt() 注入到 System Prompt 的动态段落中。

MEMORY.md 入口文件

MEMORY.md 是记忆系统的索引文件，有严格的大小限制：

  • 最多 200 行（超出部分被截断并显示警告）
  • 最多 25,000 字节（约 125 字符/行 × 200 行）
  • 每行是一个指向具体记忆文件的指针，格式：- [标题](file.md) — 一行摘要

具体记忆内容存储在独立的 .md 文件中，通过 frontmatter 标注类型：

---
name: 记忆名称
description: 一行描述（用于判断相关性）
type: user | feedback | project | reference
---

记忆内容...

记忆类型

类型内容何时写入
user用户的角色、目标、偏好、知识背景了解到用户信息时
feedback用户对工作方式的指导（避免什么、保持什么）用户纠正或确认某种做法时
project项目进展、决策、截止日期等非代码信息了解到项目背景时
reference外部资源的位置指针（Linear 项目、Grafana 看板等）了解到外部系统信息时

什么不该存入记忆： 代码模式、架构、文件路径、git 历史——这些可以通过读代码获取。记忆只存储无法从当前项目状态推导出的信息。

Session Memory

Session Memory 是会话内的结构化笔记，由后台 Agent 定期更新，在会话恢复时注入上下文。存储在 ~/.claude/projects/{project-hash}/ 目录下。

笔记结构固定，包含以下章节：

  • Session Title — 5-10 词的信息密集标题
  • Current State — 正在进行的工作、待完成任务、下一步
  • Task specification — 用户要求构建什么、设计决策
  • Files and Functions — 重要文件及其作用
  • Workflow — 常用命令及执行顺序
  • Errors & Corrections — 遇到的错误、失败的方案
  • Learnings — 有效/无效的做法
  • Worklog — 每步操作的简短记录

更新由专用 prompt 驱动，严格要求：不修改章节标题、不修改斜体描述行、只更新内容部分、所有 Edit 调用并行执行。

上下文压缩（Compact）

当对话上下文接近模型限制时，Claude Code 自动触发压缩，用摘要替换历史消息，保持会话连续性。

触发时机

  • 自动触发（autocompact）：token 用量超过阈值时，在下一次 API 调用前自动压缩
  • 手动触发：用户执行 /compact 命令
  • 微压缩（microcompact）：针对单条超大消息的轻量压缩，不影响整体历史
  • 响应式压缩（reactiveCompact）：收到 API 的 prompt_too_long 错误后立即触发

压缩过程

① 生成摘要
  • 启动专用压缩 Agent（querySource: 'compact'）
  • 发送 Compact Prompt，要求输出 <analysis> + <summary> 结构
  • 明确禁止调用任何工具（CRITICAL: Do NOT call any tools）
② 格式化摘要
  • formatCompactSummary() 去除 <analysis> 草稿块（仅用于提升摘要质量，无信息价值）
  • 将 <summary> 标签替换为可读的章节标题
③ 替换消息历史
  • 在消息历史中插入 compact boundary 标记
  • 摘要作为 user 消息注入，后续 Agent 循环从此处继续
  • 若配置了 transcriptPath，摘要中会附上完整记录的路径供查阅

部分压缩（Partial Compact）

除全量压缩外，还支持两种部分压缩模式：

  • from 模式：只摘要最近新增的消息，保留早期历史不变
  • up_to 模式：摘要早期历史，保留最近消息原文——摘要会放在最前面，后面跟着原始的新消息
