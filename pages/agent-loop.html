Agent 循环

query.ts 是 Claude Code 的心脏。每次用户输入都会触发一个循环：构建 Prompt → 调用 API → 执行工具 → 递归，直到任务完成。

循环概览

整个 Agent 循环由 query() 函数驱动，它是一个异步生成器（AsyncGenerator），向上层持续 yield 事件流（流式文本、工具调用、消息等），直到返回一个 Terminal 终止信号。

// query.ts
export async function* query(params: QueryParams): AsyncGenerator<...> {
  const terminal = yield* queryLoop(params, consumedCommandUuids)
  return terminal
}

完整数据流

点击播放，逐步演示循环过程
① 预处理
  • applyToolResultBudget() — 裁剪超大 tool result，防止 prompt 爆炸
  • microcompact() — 针对单条消息的轻量压缩
  • autocompact() — 上下文超限时触发全量摘要压缩 按需
② 构建请求
  • appendSystemContext(systemPrompt, systemContext) — 组装最终 System Prompt
  • prependUserContext(messages, userContext) — 注入用户上下文
  • 构建 API 参数：messages、tools、model、thinkingConfig 等
③ 调用 API 流式
  • deps.callModel() → 返回异步流
  • text block → 实时 yield 给 UI 渲染
  • tool_use block → 收集到 toolUseBlocks 队列
  • thinking block → 透传（需遵守 thinking 三条铁律）
④ 执行工具
  • canUseTool() 权限检查 → 通过 / 拒绝 / 弹窗询问
  • 工具执行：runTools() 或 StreamingToolExecutor
  • 生成 tool_result 消息追加到 messages
⑤ 判断是否继续
  • 有 tool_use → 回到 ① 继续下一轮（绿色虚线箭头）
  • 无 tool_use → 返回 Terminal，循环结束
  • 达到 maxTurns → 强制终止

QueryParams 入参结构

type QueryParams = {
  messages: Message[]          // 当前对话历史
  systemPrompt: SystemPrompt   // 系统提示词
  userContext: { [k: string]: string }    // 注入用户上下文
  systemContext: { [k: string]: string }  // 注入系统上下文
  canUseTool: CanUseToolFn     // 权限检查函数
  toolUseContext: ToolUseContext
  fallbackModel?: string       // 降级模型
  querySource: QuerySource     // 来源标识（repl/agent/compact...）
  maxTurns?: number            // 最大递归轮数
  taskBudget?: { total: number } // token 预算
}

循环内部状态

每次迭代开始时从 state 对象解构，continue 时整体替换，避免多个变量单独赋值：

type State = {
  messages: Message[]
  toolUseContext: ToolUseContext
  autoCompactTracking: AutoCompactTrackingState | undefined
  maxOutputTokensRecoveryCount: number  // 超 token 恢复计数
  turnCount: number
  pendingToolUseSummary: Promise<...> | undefined
  transition: Continue | undefined      // 上一次迭代的 continue 原因
}

工具执行机制

工具执行有两种模式，由 feature flag streamingToolExecution 控制：

模式实现特点
标准模式runTools()等待流式响应完成后批量执行所有工具
流式模式StreamingToolExecutor流式响应中一旦工具参数完整即立即执行，降低延迟

每个工具执行前都经过 canUseTool() 权限检查。若用户拒绝，循环会生成一条错误 tool_result 继续，而不是中断。

错误恢复机制

循环内置多种错误恢复路径：

  • max_output_tokens：输出 token 超限时，最多重试 MAX_OUTPUT_TOKENS_RECOVERY_LIMIT（3次）次，每次减少请求的 max_tokens
  • prompt_too_long：触发 autocompact 压缩后重试；若压缩失败则返回 blocking_limit 终止
  • Streaming fallback：流式调用失败时，向已有的 assistantMessages 发送 tombstone（墓碑消息）让 UI 清除，然后用非流式重试
  • FallbackTriggeredError：主模型失败时降级到 fallbackModel 重试

Thinking 模式规则

三条铁律（代码注释原文称之为"wizard rules"）：

  1. 包含 thinking 或 redacted_thinking block 的消息，所在请求的 max_thinking_length 必须 > 0
  2. thinking block 不能是消息的最后一个 block
  3. thinking block 必须在整个 assistant trajectory 中保持完整——包括同一轮的 tool_use block、后续的 tool_result，以及下一条 assistant 消息
