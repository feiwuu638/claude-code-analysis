Agent 群集（Swarm）

Claude Code 支持生成子 Agent 并行处理独立任务，通过 Coordinator 模式协调多 Agent 协作，实现复杂工作流的自动化编排。

架构概览

Agent 群集由三个层次组成：

层次角色关键文件
主 Agent接收用户指令，决策何时生成子 Agentquery.ts
Coordinator多 Agent 协调器，管理 Team、分配任务coordinator/coordinatorMode.ts
子 Agent（Worker）执行具体任务，结果返回给父 Agenttools/AgentTool/runAgent.ts

子 Agent 生成机制

主 Agent 通过调用 Agent 工具生成子 Agent。每个子 Agent 本质上是一个独立的 query() 调用，拥有自己的：

  • 独立的消息历史（从空白开始，或继承 fork 上下文）
  • 独立的工具集（可通过 tools / disallowedTools 限制）
  • 独立的 System Prompt（通过 enhanceSystemPromptWithEnvDetails() 注入环境信息）
  • 独立的 abort controller（可被父 Agent 取消）
  • 独立的 MCP 连接（按需连接子 Agent 需要的 MCP server）

Fork 与 Subagent 的区别

当 feature flag FORK_SUBAGENT 启用时，Agent 工具支持两种模式：

模式调用方式上下文适用场景
Fork不指定 subagent_type继承父 Agent 完整对话上下文和 prompt cache研究类任务、需要父 Agent 背景的工作
Subagent指定 subagent_type全新上下文，需在 prompt 中提供完整背景独立任务、专用 Agent 类型

Fork 的优势： 继承父 Agent 的 prompt cache，不需要重新缓存 System Prompt，成本更低。但不同模型的 fork 无法复用缓存，因此 fork 不应指定 model 参数。

Coordinator 模式

通过环境变量 CLAUDE_CODE_COORDINATOR_MODE=1 启用。Coordinator 是一个特殊的主 Agent，专门负责任务分解和多 Agent 协调：

  • 使用精简版 Agent Tool prompt（去掉冗余使用说明）
  • 可使用 TeamCreate / TeamDelete 工具管理 Agent 团队
  • 可使用 SendMessage 向已有 Agent 发送消息
  • Worker Agent 不能再生成 Coordinator 级别的子 Agent（防止无限嵌套）

后台 Agent

Agent 工具支持 run_in_background: true 参数，让子 Agent 在后台运行：

  • 主 Agent 不阻塞，可继续与用户交互
  • 子 Agent 完成后通过通知机制告知主 Agent
  • 主 Agent 不应轮询或主动检查后台 Agent 状态
  • 后台 Agent 的输出文件路径在工具返回结果中提供

Worktree 隔离

通过 isolation: "worktree" 参数，Agent 可以在独立的 git worktree 中运行：

  • 自动创建临时 worktree，Agent 在其中操作文件
  • 若 Agent 未做任何修改，worktree 自动清理
  • 若有修改，返回 worktree 路径和分支名，供主 Agent 决定如何合并

Agent 内存与状态

tools/AgentTool/agentMemory.ts 管理子 Agent 的状态持久化：

  • 子 Agent 的对话历史可序列化到磁盘，支持 /resume 恢复
  • SendMessage 工具通过 Agent ID 或名称找到已有 Agent，注入新消息继续执行
  • Agent 的 file state cache 从父 Agent clone，避免重复读取文件
